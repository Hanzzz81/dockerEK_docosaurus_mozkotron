💻 Cowork session ▼
Editor otevře Cowork + připojí Synology folder

Jak to funguje

  • 💻 Editor otevře Cowork desktop na svém počítači
  • 📁 Připojí sdílený Synology folder (XPR0_docker_mozkotron/docs/)
  • 📋 Claude automaticky přečte manifest z _system_work/CLAUDE.md
  • 🔄 Manifest definuje intake dialog — Claude se ptá strukturovaně

Proč Synology?

Sdílený folder = všichni editoři pracují nad stejnými daty. Změny se propagují okamžitě. Žádné merge konflikty — jeden editor, jeden dokument, jedna session.

Konzistence

Jeden manifest na Synology = stejný workflow pro všechny editory. Změna manifestu → změna dialogu pro všechny při další session. Centrální řízení bez nutnosti školení.

Co je potřeba

  • ✅ Cowork desktop nainstalovaný na počítači editora
  • ✅ Synology Drive klient připojený k XPR0_docker_mozkotron
  • ✅ Přístupová práva read/write na docs/
📋 Manifest (CLAUDE.md) ▼
Sdílená pravidla — vstupní brána pro Claude

Co manifest obsahuje

  • 📌 Intake dialog — strukturované otázky, které Claude klade editorovi (téma, kategorie, audience, podklady)
  • 📏 Frontmatter kontrakt — 10 povinných polí + 1 podmíněné (source_bimg)
  • 🔀 Rozhodovací strom vrstev — kdy BIMG, kdy SOP, kdy CHL
  • 📊 Odkaz na _structure.json — registr AOR oblastí a tematických bloků
  • ⚖️ Constrainty formátu — limity délky per vrstva
  • ✅ Checklist před publikací — 8 bodů validace

Kde leží

docs/_system_work/CLAUDE.md — Docusaurus složky s podtržítkem ignoruje, takže se nezobrazuje ve wiki. Claude ho čte automaticky na začátku každé session.

Princip

Manifest je jediný zdroj pravdy pro chování Clauda. Když Jan změní manifest, všichni editoři se při další session chovají podle nových pravidel — žádné přeškolování, žádné individuální nastavení.

Trigger — potřeba identifikována
Manažer rozhodl: „Tohle téma potřebuje dokumentaci"
Manažer (editor)

Co se děje

Manažer (v roli editora) vyhodnotí podnět z kroku 0 a rozhodne, že téma potřebuje novou nebo aktualizovanou dokumentaci. Nemusí být odborník na dané téma — stačí identifikovat potřebu.

Kdo je „Manažer (editor)"?

Jedna osoba v dvojroli:
  • Manažer = funkce ve firmě (Jan, vedoucí, ...)
  • Editor = role v tomto systému (ten, kdo zadává obsah Claudovi)
V praxi: Jan Kutílek nebo kdokoli s přístupem do Cowork + Synology, kdo má znalost tématu.

Výstup

Jasné rozhodnutí: „Toto téma potřebuje dokumentaci." Spouští intake dialog v Cowork session.

🏷️
Kategorizace
Je to LAW, PROC, KNOW, nebo MEM?
Manažer (editor)

Čtyři kategorie — čtyři pohledy na téma

  • 📜 LAW — Co platí? Závazné pravidlo, mantinel. Porušení = problém.
  • 🔧 PROC — Jak to udělat? Opakovaný postup, procedura. Dělá se pravidelně.
  • 💡 KNOW — Proč to tak je? Expertní znalost, kontext. Potřeba pro rozhodování.
  • 📝 MEM — Co se změnilo? Žurnál rozhodnutí a událostí.

Rozhodovací strom

  • 1. Porušení = problém? → LAW
  • 2. Opakuje se to pravidelně? → PROC
  • 3. Potřeba pro rozhodování? → KNOW
  • 4. Stalo se jednou, důležité zapsat? → MEM

Proč to dělá člověk a ne stroj?

Kategorie je věcné rozhodnutí o tom, CO informace je. Claude pomáhá s rozlišením (viz Průvodce pruvodce-kategorie.html), ale finální slovo má manažer — on zná kontext firmy.

Pozor: kategorie ≠ vrstva

Kategorie (LAW/PROC/KNOW/MEM) říká CO to je. Vrstva (BIMG/SOP/CHL) říká JAK to zdokumentovat. Jsou to dvě nezávislé osy — ze stejného LAW pravidla může vzniknout BIMG i SOP i CHL.

Vstupy + sběr artefaktů
Téma, znalost, audience, podklady — vše od jednoho člověka
Manažer (editor)

Co editor dodá (povinné)

  • 📌 Téma — o čem to je, co chce zapsat
  • 🧠 Znalost — co o tom ví, co je rozhodnuté, jaká je praxe
  • 👤 Audience — pro koho to je (AOR oblasti: firma, prodej, sluzby, ...)

Co editor nahraje (artefakty)

  • 📄 Dokumentace — existující postupy, manuály, specifikace
  • 📸 Fotodokumentace — fotky produktů, pracovišť, zařízení
  • 🖥️ Screenshoty — obrazovky z Money S4, e-shopu, systémů
  • 📝 Přeložený postup — textový popis „jak to dělám" vlastními slovy
  • 🎥 Video — nahrávka procesu, ukázka
  • 🔗 Odkazy — na existující docs, webové zdroje

Co editor NEŘEŠÍ

  • ❌ Typ vrstvy (BIMG/SOP/CHL) — to rozhodne Claude
  • ❌ Formátování a strukturu — constraintováno šablonami
  • ❌ Kam uložit — Claude najde správný AOR blok
  • ❌ Psaní textu — to dělá Claude

Princip R006

Editor dodává potřebu a vstupy. Parametry dokumentu (typ, formát, umístění, metadata) jsou systémová pravidla aplikovaná strojově. Editor nevolí — to vyplývá z rozhodovacího stromu.

Rozhodnutí o vrstvách
BIMG vždy. +SOP? +CHL? Návrh + konfirmace
Claude

Pravidlo R008 (strojové)

  • Téma nemá žádnou dokumentaci → vždy nejdřív BIMG
  • Existuje BIMG + konkrétní postup pro roli → navrhnout SOP
  • Existuje BIMG + SOP + potřeba rychlé reference → navrhnout CHL

Konfirmace

Claude navrhne, manažer potvrdí. Příklad: „Téma Reklamace nemá BIMG — vytvořím BIMG (⚖️ LAW). K tomu navrhuji i SOP pro roli prodejce. Souhlasíš?"

Dvouosý model (R011)

  • Category (CO to je): LAW | PROC | KNOW | MEM — z DOS taxonomie
  • Layer (JAK je to zdokumentované): BIMG | SOP | CHL
Osy jsou nezávislé — z LAW pravidla může být BIMG (kompletní popis) i SOP (postup dodržování) i CHL (quick ref lhůt).

Constraint A — Formát
Limity délky a formy pro vrstvu
Systém

Formátové limity per vrstva

  • BIMG — text + diagramy (Mermaid), subpages povolené, neomezená délka. Admonitions: :::info, :::tip, :::warning, :::danger
  • SOP — bez limitu délky, řídí se tématem. Číslované kroky, screenshoty, volitelně diagram. Žádné subpages.
  • CHL — 1×A4 tisknutelný. 3 povolené formáty: tabulka, checklist, Mermaid diagram. Žádný souvislý text. Max 1–2 screenshoty.

Princip

BIMG pokrývá téma do hloubky. SOP popisuje postup per role — délka se řídí tématem (R018). CHL se nalepí vedle monitoru — zkrácená verze nad obsáhlejším SOP.

Šablony

sablona-bimg.md · sablona-sop.md · sablona-chl.md v _system_work/

Constraint B — Meta + naming
Frontmatter, pojmenování, umístění
Systém

Frontmatter kontrakt (R016)

Vždy povinné (10 polí):
  • title (s emoji), sidebar_label, sidebar_position, description
  • category (law|proc|know|mem), layer (bimg|sop|chl)
  • audience (array z AOR: firma|management|finance|rust|nakup|prodej|sluzby|technologie)
  • version, author, date
Podmíněně povinné:
  • source_bimg — slug mateřského BIMG. Povinné pro SOP a CHL, neuvádí se pro BIMG.

Naming konvence (R003)

  • Soubory: kebab-case-bez-diakritiky.md
  • Přípona vrstvy: reklamace-bimg.md, reklamace-sop.md, reklamace-chl.md
  • Složky: stejné pravidlo, bez diakritiky

Umístění

Dle _structure.json — 8 AOR oblastí, 27 tematických bloků. Claude najde správný blok podle audience + téma.

Kaskádová revize
Změna BIMG → revize SOP → CHL
Systém

Mechanismus

  • Při změně BIMG → Claude hledá soubory s source_bimg: <slug>
  • Najde všechny SOP a CHL, které odkazují na změněný BIMG
  • Přegeneruje je s novými informacemi
  • Předá k review manažerovi

Princip

Směr je vždy shora dolů: BIMG (zdroj pravdy) → SOP (postup) → CHL (reference). Nikdy naopak. Pokud se změní SOP, nemění se BIMG — SOP je derivát.

Slug reference (R015)

source_bimg: reklamacni-rad-bimg — slug = název souboru bez .md. Claude ho používá pro vyhledávání derivátů.

Zpětná vazba
Chyba, nejasnost, chybí obsah → nový podnět → smyčka se uzavírá
Čtenář

Typy zpětné vazby

  • ❌ Chyba — „Tady je špatně uvedený termín/postup"
  • ❓ Nejasnost — „Nerozumím kroku 3"
  • ➕ Chybí obsah — „Tady by mělo být i jak řešit X"

Kam feedback jde

Feedback → dashboard manažera → manažer vyhodnotí → nový podnět (krok 0) → smyčka se uzavírá.

Proč je to důležité

Dokumentace je živý systém. Bez zpětné vazby stárne a přestává odpovídat realitě. Feedback loop zajišťuje, že se dokumentace neustále zlepšuje na základě reálného používání.
